# 03 · Rule-constructor — как правила рождаются и закрепляются

Движок — **не «решённый оптимизатор», а конструктор правил.** Жизненный цикл:

1. **Сбор максимума.** Читаем всё, чем управляем и что меряем: Advantage+, CBO/ABO, bid
   strategy (lowest_cost / cost_cap / roas_goal), cost cap / ROAS goal, событие оптимизации,
   окно атрибуции, плейсменты + метрики (spend/reach/frequency) + backend (purchases / LTV /
   refunds / stock). *(intake — `engine/schema.py:AdSetConfig` + `engine/meta_api.py`)*
2. **Кандидат-правило.** Гипотеза: «при таком сочетании сигналов делать то-то».
3. **Тест.** Правило проверяется (shadow / backtest): стало ли лучше?
4. **Промоут.** Прошло проверку (**Python**: «результаты лучше», или **человек**: «лучше») →
   правило закрепляется в **постоянной Python-иерархии** (`engine/policy.py:decide`). Пул
   «на тесте» обновляется. По мере подтверждения правил в микс добавляются новые сигналы.

Это тот же приём, что в стадии [01](../01-client-discovery/README.md): `patterns.json` +
`learn.py` — накопление одобренных правил. Здесь — то же для UA-execution.

## Честная граница

**Оптимальная ПОСЛЕДОВАТЕЛЬНОСТЬ рассмотрения всех аргументов — вопрос исследования.**
Комбинаторику `Advantage+ × CBO/ABO × bid × окно атрибуции × вертикаль` за 1.5 дня не решить и
не выдать за решённое. Что есть сейчас в коде: **сбор максимума данных + простые модели**
(вкл / выкл / изменить, при прочих равных). Это намеренно.

## Разделяем по стадиям (явно)

| Стадия | Про что |
|---|---|
| **как включаем** | запуск / возобновление ad set, бюджет, статус |
| **как создаём** | новые ad set / креативы, когда текущих не хватает |
| **какие правила применяем** | 5 промоутнутых правил (ниже) + кандидаты на тесте |
| **как оцениваем результат** | Python / человек подтверждает улучшение → промоут правила |

## Что читаем (intake), но комбинируем пока частично

Advantage+ · CBO/ABO · bid strategy / cost cap / ROAS goal · окно атрибуции · optimization
event · плейсменты · reach/frequency · backend (LTV/refunds/stock). Это и есть ответ на
«учтены ли ABO/CBO/Advantage+»: **читаем — да; оптимально комбинируем — research.**

## Правила сейчас

`python engine/rules.py` печатает реестр. Кратко:

**Промоутнуты (в `decide`):** проверка невиновности · learning-hold · scale-если-бюджет-выбирается ·
креатив-диагностика (CTR↓/CVR ок) · burnout-pause.

**На тесте (кандидаты):** CBO-vs-ABO бюджет · cost-cap-breach · attribution-window-aware ·
incrementality-holdout.
