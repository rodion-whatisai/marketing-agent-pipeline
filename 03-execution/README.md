# 03 — Execution

> Двигать бюджет по живой кампании — scale / kill / hold — так, чтобы каждое движение денег
> прошло через детерминированный гейт, оставило след и откатывалось. Это **вершина** всей
> дисциплины: здесь агент впервые трогает деньги, поэтому «высокорисковое проверяет код, а
> не второй агент» перестаёт быть лозунгом.

> **Статус: спроектировано, в коде ещё нет.** Design-блок, как 02. Нет Meta API, нет
> реального аккаунта; пороги (target CPA, +20 %, длины окон) — иллюстративные, настраиваются
> под клиента. Часть рассуждений — «как я строил бы систему», а не из ручной отладки:
> помечено по тексту.

> **SCOPE: только PERFORMANCE-кампании** (оптимизация на purchases / conversions). Не
> awareness, не reach, не engagement. Всё ниже — про результат = продажа.

> **Концептуальный скелет движка** (типы · Meta-заглушки · сигналы · решение + JSON audit,
> гоняется на стабах, не live) — [`engine/`](engine/).

## Что эта стадия потребляет и выдаёт

**Вход:** портфель ad sets кампании с метриками (`spend, CPA, ROAS, purchases, CTR, CVR,
CPM, learning_status, budget, last_edit, creative_age, attribution_lag, AOV/корзина,
наличие/anchor SKU`) + цель и бюджет кампании. Пример — [mock-adsets.csv](mock-adsets.csv)
на 5 ad sets (таблица ниже).

**Выход:** по каждому ad set — решение **scale / kill / hold / → human approval** + движение
бюджета, где каждое прошло детерминированный гейт, записано в audit-лог и обратимо
(rollback). Плюс **кампанийный форкаст**: доставит ли портфель плановый результат и освоит
ли бюджет.

## Lead thesis — здесь начинаются деньги

До сих пор агент только читал и предлагал. В execution он **трогает бюджет** — самое
необратимое действие конвейера. Поэтому дисциплина «агент судит, Python гейтит, человек
апрувит» здесь не украшение, а защита от слива.

И вот что выясняется, когда раскладываешь реальную UA-логику (scale / kill / hold) на
«правило vs суждение»: **почти всё — правило.** Окна атрибуции, ранги, отношения, тренды,
пороги, форкаст — детерминированы. Агент остаётся тонким (сформулировать «почему»,
предложить в рамках гейта); человек — на необратимом (kill, крупный бюджет, ручной досмотр).
Money-moving агент — это в основном **питон-гарды с тонким агентским хвостом**, а не «умный
агент, которому доверили кошелёк».

## Граница код / агент / человек

| Часть | Кто | Почему |
|---|---|---|
| окна атрибуции (×3) + среднее, размазка спенда вперёд, взвешивание результата по дотоку | **Python** | арифметика по времени, не суждение |
| total / blended CPA·ROAS, коэффициент полезности (%продаж ÷ %бюджета) | **Python** | агрегация и отношения |
| детект «бюджет не выбирается», +20 %/шаг scale-cap, pacing | **Python** | пороги и лимиты |
| ранг-диссонанс (медиана CTR/CVR), CPA-тренд, протокол 2×5 дней, B-ранги | **Python** | ранги и тренды по таблице |
| `ROAS = AOV / CPA` декомпозиция, профиль корзины на рекламу | **Python** | разложение, агрегация |
| «проверка невиновности» (атрибуция · сайт · сток) перед kill | **Python** | детерминированный чек-лист |

| «почему» реклама тянет дешёвые / полные корзины; формулировка причины для аудита; предложение действия в рамках гейта | **агент (тонко)** | гипотеза, не правило |
| ручной досмотр сайта + Clarity-сессии (ветка диссонанс→сайт); апрув kill / крупного бюджета; ресток | **человек** | необратимое / вне рекламы |

## Decision policy — scale / kill / hold

Агент **предлагает**, детерминированное правило **гейтит**. Пороги ниже — иллюстративные.

- **Scale** — если ad set стабильно **выбирает бюджет** (см. 5-дневный спенд) и эффективен
  (CPA < target, утилити > 1) → поднять бюджет **постепенно, максимум +20 %/шаг**. Если
  бюджет **не выбирается** — управлять им бессмысленно (упёрся в delivery, не в бюджет),
  scale не поможет.
- **Kill** — низкий ROAS + высокий спенд + низкий CTR + CPA кратно над target (кейс **B**),
  **но только пройдя «проверку невиновности»** (ниже). Гранулярность: на уровне ad set →
  сперва **чинить креатив**; на уровне отдельного объявления → **выключать**.
- **Hold** — низкий спенд + learning → **не трогаем** (правка сбрасывает learning), кроме
  сильных сигналов. Протокол: **окно-1 (5 д)** без улучшения CPA-тренда → зафиксировать
  суждение, hold ещё окно; **окно-2 (+5 д)**: плоско → **kill**, вверх → **hold**, вниз →
  **kill**. Каждое суждение пишется снапшотом (как поток брифа в 02).

## «Проверка невиновности» — чек-лист перед любым kill

Сигнал «плохой CPA/ROAS» — ещё не вина рекламы. Прежде чем агент убьёт ad set, код обязан
исключить три внешние причины (execution-версия «absence of signal ≠ signal» из 01):

1. **Атрибуция** — конверсии ещё не доехали. Если бóльшая часть спенда свежая, а окно
   атрибуции не закрыто (кейс **D**: 5 дней назад показывало 0, сейчас 18) → не kill, **wait**.
2. **Сайт** — **диссонанс**: ad set в верхней половине по CTR, но в нижней по CVR. Креатив
   притянул, а на сайте не нашли / не купили. Хорошая CVR + плохой CTR → менять креатив;
   хороший CTR + плохая CVR → проблема **сайта** → человеку на ручной досмотр + Clarity.
3. **Сток** — нет в наличии **якорного SKU** (с него начинается корзина). Якорь OOS →
   корзина схлопывается, продажи вниз, реклама ни при чём → **блок kill + флаг**.

Только если все три «невиновны» — kill доходит до гейта.

## Confidence / атрибуция

Не судим свежий спенд по недозревшим конверсиям:
- смотрим данные под **3 разными окнами атрибуции**, считаем **среднее окно**;
- **размазываем недавний спенд вперёд** на окно (потраченное сегодня даёт конверсии через
  ~N дней) и **взвешиваем наблюдаемый результат на ожидаемый доток** — только потом считаем
  CPA/ROAS для решения;
- если **same-day click→purchase** (лага нет) — last-click надёжен, верим глазам.

> *Автор напрямую кейсы не «прокликивал» руками — это рассуждение о том, как он строил
> бы систему управления ad sets, а не результат ручной отладки.*

## Структура результата — корзина и сток

«Purchase» не атомарен: это **корзина из нескольких продуктов** или **один item**. Поэтому:

- **`ROAS = AOV / CPA`** — аномальный ROAS при нормальном CPA чаще про **корзину** (что
  купили), а не про стоимость. Считаем **профиль корзины на рекламу** (средний чек,
  items/корзину, микс продуктов) из заказов, привязанных к ad set.
- В mock AOV держим плоским $60 (упрощение). Если у **D** корзины крупные (AOV $120) →
  ROAS 1.44, не 0.72 — не слив; если у **A** дешёвый single-item (AOV $25) → ROAS 1.25, не
  3.0 — перескейлен. **Корзина переворачивает и kill, и scale.**
- Глубже — **маржа**: revenue ≠ profit, состав корзины бьёт по марже. *(next, не в скоупе.)*

## Portfolio + кампанийный форкаст

- **Коэффициент полезности** = %продаж ÷ %бюджета ad set. >1 — тянет выше своего веса, <1 —
  балласт (см. таблицу ниже: A = 1.79, B = 0.39).
- **Кампанийный форкаст** (guardrail уровня кампании): после правок в ad sets проверить —
  **доставит ли портфель плановый результат И освоит ли бюджет кампании**. Анти-паттерн:
  локально наоптимизировал ad sets → кампания недопотратила и недодала. Локальная оптимизация
  не должна ломать глобальную доставку.

## Guardrails — что агент НЕ имеет права

- **АГЕНТ НИКОГДА НЕ УДАЛЯЕТ** — ни кампанию, ни ad set, ни объявление, ни данные.
  «Kill» = пауза / выключение / архив, **не** удаление. Удаление необратимо и ломает и
  audit, и rollback; вся дисциплина держится на том, что прежнее состояние всегда сохранено.
- никаких движений деньгами мимо гейта; kill — только после «проверки невиновности»;
- scale — **максимум +20 %/шаг**; no-edit во время learning; spend caps (дневной / аккаунт);
- portfolio-инварианты (сумма = бюджет, нет переконцентрации, cap на клиента);
- крупное / необратимое (kill кампании, большой сдвиг бюджета, рефреш на $700/день у **E**) —
  не auto, а в **очередь человеку**.

## Human approval · audit · rollback

- **Approval layer** — детерминированный роутер: мелкое / обратимое в пределах гардов
  **auto-approve**; крупное / необратимое → **очередь человеку**.
- **Audit** — append-only запись на каждое решение: `{ts, ad_set, сигналы, какое правило
  сработало, проверка невиновности, предложение агента, вердикт гейта, действие человека,
  before/after}`. Провенанс 01 + снапшот-поток 02, применённые к деньгам: каждый доллар
  трассируется до правила и апрува.
- **Rollback** — каждое действие хранит prior state; авто-откат, если гард сработал после
  действия, или по кнопке человека.

## Backtest / shadow / policy-sim — как заслужить право двигать деньги

Та же лестница доверия, что в корневом [README](../README.md), но про реальные деньги:
- **backtest** — прогнать policy на истории: помогла бы или навредила;
- **shadow mode** — live, но только **логирует** решения, не применяет; сверка с человеком;
- **policy simulation** — прогон изменения policy против mock-датасета до деплоя.

Агент получает право на авто-движения только пройдя эту лестницу.

## Скетч движка — `engine/`

Концептуальный Python-скелет policy-движка — [engine/](engine/): типы (`schema.py`) · Meta-API
**заглушки** с реальными эндпоинтами (`meta_api.py`) · детерминированные сигналы (`signals.py`) ·
решение + JSON audit (`policy.py`). Гоняется на стабах (`python engine/policy.py`), **не на
live-аккаунте** — скетч, не прод.

Воплощает принцип **«Python переваривает → агент ест готовые сигналы»** (LLM сырьё читают плохо)
и закрывает все 9 критериев теста ревьюера (delayed attribution · premature-kill defense ·
creative/site/offer/stock · budget conservation · campaign delivery · audit · human gate ·
agent-vs-code · mock-history).

Прогон по 5 ad set (gate 1 невиновность → gate 2 learning → gate 3 эффективность):
A→`scale`, B→`send_to_human` (креатив), C→`hold` (learning), D→`hold` (атрибуция не дозрела —
wait), E→`send_to_human` (усталость креатива). `test_policy.py` пинит эти исходы.

Движок — **конструктор правил**: читаем максимум (Advantage+, CBO/ABO, bid strategy /

## Как CBO / ABO / Advantage+ меняют decision layer

Где живёт бюджет — определяет, что вообще можно «скейлить»:

- **ABO** (бюджет на ad set) — `scale` поднимает бюджет конкретного ad set. Дефолт в моке.
- **CBO / Advantage+** (бюджет на кампании) — Meta распределяет бюджет между ad set'ами **сама**.
  Дёргать бюджет ad set **бесполезно**: оптимизатор перельёт обратно. Двигать нужно на уровне
  **кампании** (либо менять не-бюджетные рычаги — таргет, креатив, ставку).

Поэтому при CBO эффективный ad set не уходит в ad-set-`scale`, а в **campaign-level** очередь.
Зафиксировано как **обязательное** reserved-правило (`rules.RESERVED → cbo_no_adset_budget`) и
проверяется в `policy.decide` (gate 3) перед любым scale.
