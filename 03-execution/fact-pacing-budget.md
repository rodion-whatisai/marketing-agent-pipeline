# Отчётность: план-факт → новый дневной бюджет на город (колонка FW)

> Разбор отчётного дашборда, который замыкает петлю: тянет живые результаты Meta (с
> разбивкой по регионам) → сворачивает в города → считает, как город идёт к месячному KPI
> по лидам → и выставляет **новый дневной бюджет на город**, чтобы догнать план. Формулы
> реверс-инжинирены из воркбука **«OLD dashboard formulas»** (лист `New dashboard`).
> Источники `Meta!` и `Rest!` в этой копии отрезаны (значения `#N/A`/`#REF!`), поэтому
> читали по формулам.

Это **факт-половина** того же цикла, что планинг в [02](../02-planning/daily-budget-pacing.md):
там бенчмарк → подневный план, здесь факт → коррекция бюджета. Вводим термины по ходу.

- **План-факт** — сравнение планового KPI с фактически доставленным.
- **Регион** — Meta отдаёт результаты с разбивкой по географии (провинция / штат).
- **Город** — наш юнит планирования (VAN/MIA/TOR/OTT/MTL/DAL/HOU). Регион и город — не
  одно и то же: одна провинция может кормить два города.
- **Тип лида** — `BD` (день рождения, birthday) или `Wide` (широкая аудитория). Определяется
  по имени ад-сета (`...JUNBDay` vs `...Wide`).
- **Caller** — человек, обрабатывающий лиды. Ограничивает, сколько лидов вообще имеет смысл
  лить (ёмкость обработки).
- **FW** — колонка `FW` листа, финал расчёта: новый дневной бюджет, нужный городу сейчас.

---

## 1. Принцип одной фразой

> Сколько лида ещё надо до KPI, делим на оставшиеся дни и умножаем на **текущую** цену лида.
> Получаем дневной бюджет, который догонит план при сегодняшней эффективности — отдельно
> на каждый город.

---

## 2. Поток данных

```
Meta raw (лист Meta: Region · Adset · spend · leads · clicks · date)
   │  SUMIFS по региону + по диапазону месяца
   ▼
[1] Регион-датасет   +   [2] матрица Ад-сет × Регион
   │  свод по городам (стек колонки EO = SUM по всей матрице)
   ▼
[3] Город: Total leads · Total spent · CPL
   │  + [4] сплит провинции (TOR+OTT, DAL+HOU) по доле города
   │  + [5] тип лида: %BD, %Wide
   ▼
[6] Пейсинг к KPI  →  новый дневной бюджет  (колонка FW)
```

---

## 3. Механизмы (по формулам листа)

Буквы — реальные колонки `New dashboard`.

**[1] Регион-датасет** (`B5`):
```excel
=SUMIFS(Meta!spend, Meta!Region=$A5, Meta!date>=начало_месяца, Meta!date<=конец_месяца)
```
Берёт из сырого листа `Meta` сумму по региону за месяц. Колонки источника: `Region · платформа
(ig/fb) · Adset · spend · leads · clicks · date`.

**[2] Матрица Ад-сет × Регион** (`O5`):
```excel
=SUMIFS(Meta!spend, Meta!Region=$N5, Meta!Adset=O$4, date в месяце)
```
Строки — регионы, колонки `O…EN` — имена ад-сетов. Каждая ячейка = результат ад-сета в
регионе. Это и есть «разбивка каждого ад-сета по регионам».

**[3] Свод в город** — стек колонки `EO`: `EO19 = SUM(O19:EN19)` (сумма строки матрицы = всего
по городу). Бэнды стека: лиды (стр. 18–26), spent (37–44), BD-лиды (48–56), Wide-лиды (78–86).

**[4] Сплит провинции** — когда регион кормит два города (`EY5 = "TOR+OTT"`, `EY9 = "DAL+HOU"`),
доля считается **динамически от собственных объёмов городов**, не хардкодом:
```excel
доля_города = лиды_города / (лиды_город1 + лиды_город2)     # EZ6 = FC8/(FC7+FC8)
```
Это и есть «среднее распределение»: в реальных данных даёт те самые ~76.67% / ~23.33%.

**[5] Тип лида** (`EO59`):
```excel
% BD   = BD-лиды / Total = EO49/EO19
% Wide = Wide-лиды / Total = EO79/EO19
```

**Ёмкость обработки** — `Callers × leads-per-caller`. Потолок: лить лиды сверх того, что
callers успеют обработать, бессмысленно.

## 4. Ядро — цепочка FW (по строке города)

```
Current CPL       = Total spent / Total leads          (FM = FL / FK)
To deliver        = Plan Leads − Leads факт            (FR = FO − FK)
Leads/day needed  = To deliver / дни оставшиеся         (FV = (FO − FK) / (C2 − D2))
► NEW daily budget = Leads/day needed × Current CPL      (FW = FV × FM)   ← на город
```

Плюс **сигнал пейсинга** — опережаем график или отстаём:
```
% доставлено = FK / FO         % времени прошло = D2 / C2
dif = % доставлено − % времени  (FU = FT − FS)    # + впереди, − отстаём
```

Логика FW: отстающему и дорогому городу нужен больший дневной бюджет; идущему с опережением
и дёшево — меньший. Цена лида берётся **текущая** (по факту месяца), не плановая.

показывает, кому горит.

## 6. Связь с 02 и движком

- **Замыкание петли.** В [02](../02-planning/daily-budget-pacing.md) бенчмарк раскладывался в
  подневный *план*; здесь *факт* из Meta сверяется с KPI и пересчитывает бюджет. Один и тот же
  город, две стороны цикла.
- **FW = живой бюджетный сигнал.** «Новый дневной бюджет на город» — ровно то, что в этой
  стадии движок выдавал бы сам: отстаём по KPI → вот сумма под текущий CPL. Read-only insights
  → pacing → рекомендация бюджета — это первая, безопасная ступень roadmap движка (см.
  [README](README.md), раздел Roadmap), до любых write-действий.
- **Гейт по ёмкости.** `Callers` — тот же тип ограничения, что budget-гард движка: не лить
  лиды, которые некому обработать.
