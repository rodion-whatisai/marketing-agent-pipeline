# 01 — Client Discovery

> Найти и квалифицировать клиента по жёстким проверяемым сигналам — тратит ли он на
> рекламу и сломан ли у него трекинг — и зафиксировать это в железобетонные факты, с
> которыми можно зайти.

> **Статус:** наполнено (Шаг 3) — эталонный блок. Реальный рабочий код, реальные
> прогоны (см. [Evidence](#evidence--реальные-прогоны)).

> Упоминания кода ниже — **кликабельные ссылки на реальные файлы этого репозитория**.

## Что эта стадия потребляет и выдаёт

**Вход:** домен или популяция доменов.

**Выход:** не просто набор фактов и не «отчёт, классный сам по себе» — а
**валидированный отчёт, который можно сразу отправить клиенту как первое касание
(outreach)**.

Это и есть цель стадии. Не «сделать классный отчёт», а **отправить классный — то есть
правильный, проверенный, с провенансом и дисклеймерами** (см. секцию «Guardrails и
целостность данных» ниже). «Валидированный» значит: каждый факт датирован, помечен как
добыт и проверяем по ссылке на источник — такой отчёт не стыдно и не опасно слать, и он
**несёт клиенту value ещё до первого разговора**. В обычном аутриче так не делают.

Python здесь — источник правды для всех агентов ниже по конвейеру:
сигнал → валидация → железобетонный факт → отчёт → outreach.

## Lead thesis — история лида

Client discovery — это **поиск по сигналам и паттернам**. Наводишь движок на популяцию
сайтов; детерминированные правила выцепляют тех, кто подходит под профиль.
Перенастраиваешь — меняя правила, не код.

Профиль в этом примере намеренно узкий: **мелкие бизнесы** (агентство-на-одного,
локальный магазин), которые *только начали* лить на Meta, но трекают это неправильно —
живой пиксель, который не стреляет конверсию; checkout без `AddToCart`; бюджет, уходящий
в слепую воронку. Тратят → до них есть смысл достучаться; трекинг сломан → есть что
конкретно продать; мелкие → решение быстрое. Эта комбинация = **tier-1 лид**.

Перенастроенный на другое правило (скажем, порог активных объявлений на страну) тот же
движок находит другую популяцию. *Кто считается лидом — это число в правиле, а не
ощущение.*

## Граница код / агент

Во всём discovery агент (Claude Haiku) работает **ровно в одной точке** —
[`page_classifier.py`](../../page_classifier.py), и только на тех URL, которые
детерминированная лестница не распознала.

| Часть | Код или агент | Почему |
|---|---|---|
| sitemap, детект платформы, перехват пиксель-событий, поиск FB/Google рекламы, fuzzy-match, отчёт | **детерминированный Python** | проверяемо, повторяемо, дёшево; ноль LLM-вызовов |
| смысл-тип URL — только хвост, который лестница правил не поймала | **агент (Claude Haiku)**, батч по 50 | семантика slug'а — суждение, не правило |
| ожидаемые события для типа · провенанс `method=` · квалификация лида | **детерминированный Python** | огорожено после/вокруг агента |

Лестница (дешёвое и детерминированное первым, агент — только на остаток):

```python
# page_classifier.py (сокращено)
def classify_urls(urls, platform=""):
    for url in urls:
        fast = fast_classify(path)         # 1) patterns.json → regex FAST_RULES
        if fast: ...                       #    распознано правилом — агент не нужен
        if platform == "shopify" and ...:  # 2) Shopify slug-правила (бесплатно)
            ...
        ai_needed.append((i, path, url))   # 3) только нераспознанное → Claude Haiku

    # тип решает агент — но КАКИЕ события на нём ждать навешивает
    # детерминированный словарь УЖЕ ПОСЛЕ агента, плюс провенанс-штамп:
    clf["expect_events"] = _get_expect_events(clf["type"])
    clf["method"] = "claude"               # видно: это решил агент, а не правило
```

→ живой код: [`classify_urls`](../../page_classifier.py#L458) (лестница) ·
[`_get_expect_events`](../../page_classifier.py#L423) (огораживание агента) ·
[`FAST_RULES`](../../page_classifier.py#L139) (детерминированные паттерны).

Агент решает только *тип* (ярлык). Что с этим типом делать — какие конверсионные события
на нём положено увидеть — навешивает детерминированный словарь `_get_expect_events`
**после** агента. А `method=` помечает, кто принял решение (`patterns_json` / `regex` /
`claude` / `no_api_key` / …). Любой сбой агента деградирует в безопасный дефолт
(`general`), прогон не падает.

> **Сноска:** чтобы покрыть больше кейсов, где-то понадобятся ещё агенты — но логика их
> подчинения та же (узкая задача, огорожен детерминированным кодом, продукт →
> следующему, человеческий гейт). Не «навсегда один агент», а «сколько угодно агентов
> под той же дисциплиной».

## Инженерные решения — и как мы к ним пришли

**Накопительная база паттернов ([`patterns.json`](../../patterns.json) + интерактивный
[`learn.py`](../../learn.py)).** Прогонов будет очень много (тысячи сайтов), а чистый
LLM-классификатор жрёт токены на каждом URL. База паттернов пополняется через ручной
апрув: каждый URL, пойманный `patterns.json`/regex — это URL, за который не платишь
Claude. Отмели чисто-regex (не тянет семантику незнакомых slug'ов) и чисто-LLM (дорого на
масштабе). Лестница — компромисс: детерминированное берёт большинство, агент — только
хвост.

**Стратегия Facebook Ads Library.** Пришли перебором, пошагово: поиск по правильному
имени бренда → простой поиск по ID → комбинация с keyword-строкой → перебор вариаций
имени. Каждый шаг — ручной инспект HTML / network / элементов, чтобы вычленить ровно
нужный сигнал (счётчик активных объявлений). Отсюда в коде
([`fb_ads_scraper.py`](../../fb_ads_scraper.py)): routing classic-vs-new-style Page ID
(для разных типов ID живы разные эндпоинты), keyword-fallback с fuzzy-match по имени,
пост-фильтр шума.

**Google Transparency Center** ([`google_ads_domain.py`](../../google_ads_domain.py) +
[`google_ads_creative.py`](../../google_ads_creative.py))**.** Логичное продолжение FB —
TC отдаёт результат чище и понятнее. Та же модель (домен → рекламодатели → креативы),
отлаженная на FB, переносится. *Next step:* добавить открытую базу объявлений ЕС (которую
ЕС обязал Google публиковать) как доп-источник для Европы.

**Симуляция журнея ([`clicker.py`](../../clicker.py)).** Пассивный скан не ловит события,
которые стреляют только по действию пользователя. Задача — проверить, можно ли
«прокликать» воронку до конверсионного события (`AddToCart` / `InitiateCheckout`),
**останавливаясь до реальной оплаты**. Для каждой страницы вручную выяснялось: какой
элемент на каком сайт-билдере по клику тригерит какое событие с какими параметрами; и
если трекер не виден в одной среде (GTM) — видно ли его на самом сайте и как это читать.

**Bucket `unverified`.** Booking-инструмент (Cal.com/Calendly) на странице — конверсия
может уходить server-side, браузер её не видит. Назвать это «дырой» = соврать. Поэтому
такие страницы идут в `unverified` («разобраться вручную»), не в `gaps`. Не объявляем
gap там, где не можем проверить.

**WAF — честная деградация.** Если сайт за Cloudflare/Akamai и не пускает ни `requests`,
ни headless-браузер — помечаем `blocked_by_waf` и рекомендуем ручную проверку.
Стелс-плагины, резидентные прокси, CAPTCHA-солверы — out of scope. Решения, наверное,
есть, но это не наша игра: честное «не достучались» полезнее клиенту, чем угаданные
цифры.

## Guardrails и целостность данных

**Высокорисковое действие на этой стадии — не правка бюджета (денег тут нет), а отправка
неверного или перезаявленного факта клиенту в первом касании.** Облажаться с аутричом —
самый дорогой провал стадии: один неверный «у вас сломан пиксель» убивает доверие до
начала разговора.

Поэтому *что вообще можно заявить* решает детерминированный **стек честности**, встроенный
в отчёт — код, а не усмотрение агента. Семь механизмов, дословно из
[`generate_site_report.py`](../../generate_site_report.py) /
[`generate_batch_report.py`](../../generate_batch_report.py):

1. **Датированный снапшот** ([дата из mtime файла данных](../../generate_site_report.py#L750)):
   > *Snapshot date: 2026-06-24 … Data was captured at this moment from Facebook Ads
   > Library … Ads Library is live … advertisers may have added new ads or ended running
   > ones since this report was generated.*

2. **Метка авторитетности самого числа** —
   [режим поиска прописан словами](../../generate_site_report.py#L554):
   > `page` → *Advertiser-filtered (authoritative — FB filtered by page_id)*
   > `keyword_filtered_by_name` → *fuzzy filter on advertiser name (lower confidence)*
   > `keyword_raw` → *Keyword search, unfiltered (may include unrelated ads)*

3. **Раскрытие шума** (raw vs matched):
   > *Filtered out: {N} keyword matches from other advertisers (noise)*

4. **Warning при низкой уверенности:**
   > *⚠️ Lower-confidence result. We used keyword search + fuzzy-matched advertiser names
   > … Some matches may not actually be from this brand. Manual verification recommended.*

5. **Провенанс на каждом факте identity:**
   > *Platform — {confidence} confidence · Language — detected via {source} · Country —
   > detected via {source}*

6. **Честный WAF-out-of-scope** ([batch-отчёт](../../generate_batch_report.py#L649)):
   > *…we mark the site as `blocked_by_waf` and recommend manual verification. We do not
   > use stealth plugins, residential proxies, or captcha solvers — these are explicitly
   > out of scope.*

7. **Независимая проверка вшита в каждый факт** — кнопка:
   > *Verify yourself in Ads Library →* (ведёт на живую страницу-источник)

Факт покидает стадию только с прицепленными «когда это было верно» и «как добыто». Это
discovery-аналог kill switch: на языке operating model — **trust in data, measurability**.
Все семь видны вживую в реальных отчётах ниже.

(Плюс: статусы tracking-скана `OK / GAP / NO TRACKING / No CTA` никогда не смешиваются —
каждый нарратив отдельно.)

## Next step — валидатор отчёта (пока нет)

Отчёт генерится и помечается механизмами честности выше, но **на смысловую связность
перед отправкой пока не проверяется**. Пример: в демо `finom.co` креативы идут на
нескольких языках (FR/DE/IT/…) — нормально для пан-европейского рекламодателя, но тул это
никак не комментирует.

Недостающее звено — **валидатор**: проход (Python, или агент → Python) перед тем как
отчёт считается «готовым к отправке»:
- *детерминированно (Python):* счётчики сходятся, дисклеймеры на месте, снапшот свежий,
  объявления вообще есть;
- *суждение (агент):* «креативы на N языках», «оффер не похож на заявленный профиль»,
  «бренд в Ad Library не тот» → **флаг**; пас/стоп решает детерминированное правило.

Тот же принцип, что и везде: агент **флагует**, детерминированный гейт **решает**.
Валидатор — естественный следующий агент под той же дисциплиной подчинения.

> Статус: **планируется**, в коде ещё нет — зафиксировано как недостающее звено между
> «отчёт сгенерён» и «отчёт можно слать».

## Evidence — реальные прогоны

Три реальных скана, **как есть** (не превью). Отчёты сгенерены тулзой
([`generate_site_report.py`](../../generate_site_report.py)) из захваченных `fb.json`;
креативы встроены в HTML (base64, файл self-contained — открывается локально, ничего не
подгружает). Разный масштаб рекламодателя:

| Демо | Активных объявлений | Отчёт | Сырые данные |
|---|---|---|---|
| **Большой** | finom.co — **188** | [report.html](evidence/finom.co/report.html) | [fb.json](evidence/finom.co/fb.json) |
| **Средний** | semrush.com — **53** | [report.html](evidence/semrush.com/report.html) | [fb.json](evidence/semrush.com/fb.json) |

На каждом отчёте видны все семь механизмов честности из секции выше: датированный
снапшот, метка режима/авторитетности, raw-vs-matched, кнопка *Verify yourself in Ads
Library*. `fb.json` рядом — сырые захваченные данные с провенанс-полями (`discovery_method`,
`ads_library_mode`, `raw_keyword_total`, `homepage_fetch_method`); это и есть «запись
прогона» (батч не пишет отдельный консольный лог на домен).
