# CPM-бенчмарки — таблица `benchmarks` в базе фабрики

Источник истины — **таблица `benchmarks` в Postgres фабрики**
(`~\tnc-factory`, база `factory`). Это решение D12
из `FACTORY.md`: «Бенчмарки CPM/CTR/CVR — таблица с провенансом».

## Как править руками

1. База поднята: `cd ~\tnc-factory && docker compose up -d`
2. Открыть **http://localhost:8080** (Adminer — веб-таблица в браузере):
   - System: **PostgreSQL**, Server: **db**
   - Username / Password / Database — из `.env` фабрики (`factory`)
3. Выбрать таблицу **benchmarks** → **Select data** → правишь ячейки мышью.

Google Sheets больше не используется. Прежняя таблица в Drive
(«TNC Pitch-Adviser — CPM бенчмарки v2») — архив, можно удалить.

## Устройство таблицы

Одна строка = один профиль кампании с готовой вилкой значения.

| Колонка | Что это |
|---|---|
| `param` | `CPM` / `Частота при небольшом охвате` / `Частота при большом охвате` / `Порог небольшой/большой охват, людей` |
| `objective` | `Sales (Conversions)` или `Traffic / Engagement` |
| `targeting` | вертикаль: «Онлайн-образование / курсы», «БЕЗ таргетинга (вся аудитория, broad)» и т.д. |
| `geo` | `Worldwide` или `WW-чистый` (без дешёвых стран) |
| `gender`, `age_from`, `age_to`, `placements`, `formats` | соцдем и площадки |
| `value_from`, `value_to` | сама вилка |
| `comment` | пояснение словами («чистый WW × 1.2») |
| `provenance` | откуда число (требование D12) |
| `status` | `placeholder` — прикидка из ресёрча, `verified` — Родион взвесил |

Состав (116 строк): 4 якоря без таргетинга + 9 вертикалей × 2 objective ×
6 комбинаций (WW-всё, WW-чистый, только FB ×1.2, только IG ×1.1,
один пол ×1.25–1.5, возраст 18–30 ×1.15–1.3) + 4 служебные строки.
Только Meta. Awareness не используем.

Обоснования и источники стартовых цифр — в
[cpm-research-notes.md](cpm-research-notes.md).

## Правило выбора бенчмарка для адвайзера

- На сайте клиента есть вменяемое конверсионное событие → его кампании
  считаем **Sales** → берём Sales-строки.
- Событий нет или нет пикселя → **Traffic** → Traffic-строки, и это явно
  пишем в документе (вводная приходит из site-репорта сканера).

## Технические детали

- Читает бенчмарки только `benchmarks_loader.py` (через psycopg).
- После каждого удачного чтения содержимое дублируется в `cpm-cache.csv` —
  если база не поднята, расчёт всё равно идёт, с предупреждением о дате кэша.
- Первичная заливка строк: `python db/seed_benchmarks.py` в репо фабрики
  (идемпотентно, строки со статусом `verified` не перезатирает).
