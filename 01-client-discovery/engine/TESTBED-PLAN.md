# Испытательный стенд для сканера: как выйти из бесконечного whack-a-mole

## Контекст — почему это делаем

Сейчас процесс такой: берём новый сайт → находим новую очевидную дыру (не видим кнопки, фейковый Snapchat, редиректы) → чиним → следующий сайт → новая дыра. Ощущение, что это никогда не сойдётся, и возникает соблазн переписать всё с нуля.

**Разведка по репо (2 агента, факты проверены) дала два вывода:**

1. **Переписывать не надо.** Ядро пайплайна (step1 → определение платформы → скан → отчёт) архитектурно здоровое. Вся гниль сконцентрирована в **слое детекции**: знание о пиксель-платформах размазано по ТРЁМ несинхронизированным таблицам (`PIXEL_RULES` в base_scanner, Shopify-маркеры в shopify_scanner, `PLATFORM_SIGNATURES` в gtm_analyzer — там даже ключ "Hotjar" объявлен дважды и второй молча затирает первый). Pinterest нет ни в одной, Snapchat — только в одной. Плюс substring-матчинг без границ слов (отсюда Cal.com из "tetralogiCAL.COM"). ~80% повторяющихся багов — из этих мест.

2. **Твоя интуиция («прогнали сайт → додрочили → следующий → после 10 паттерны сходятся») — правильная, но в ней не хватает трёх деталей**, без которых сходимости не будет:
   - **правда нигде не записана** — после «додрочили» никто не фиксирует «вот что сканер ОБЯЗАН видеть на этом сайте»;
   - **старые сайты не перепрогоняются** — фикс для сайта №7 может молча сломать сайты №1–6, и никто не заметит (это называется **регрессия**);
   - **нет числа, показывающего прогресс** — «доверять можно» сейчас определяется ощущением, а не метрикой.

Решение всех трёх — **испытательный стенд**: замороженный набор из 11 сайтов с записанными правильными ответами («**эталон**», golden corpus), который автоматически перепрогоняется после каждого изменения и выдаёт счёт «N из M совпало». Развитие сканера превращается из «тушим сегодняшний пожар» в «двигаем число к 100%».

В репо уже есть почти всё для этого: 17 доменов с полными результатами step1+step2, ручные вердикты «10/12 MATCH» в shakedown README, шаблон раннера (mass_run_creatives.py), шаблон scorecard-CSV (smoke_test_creatives.py), шаблон «предложи → человек апрувнул → сохранили» (learn.py). Не хватает только формата эталона, инструмента сравнения и самого раннера.

## Комиссия (разные точки зрения, как просил)

**Project Manager:** «Done» должно быть числом, не ощущением. Критерий доверия: 3 новых сайта подряд одного класса (например Shopify), где триаж не нашёл ни одного дефекта кода — только пополнение эталона, — при счёте стенда ≥95%. До этого числа сканеру доверять нельзя, после — можно, по этому классу платформ. Google в сканер добавлять ТОЛЬКО после того как стенд зелёный — иначе новые баги нельзя будет отличить от старых.

**Engineer:** главный структурный дефект — три параллельные таблицы платформ, которые надо синхронизировать руками. Это единственный рефакторинг, который стоит делать на этой неделе: один реестр `platforms.py`, из которого все три потребителя читают. Но делать его СТРОГО после того, как стенд построен — чтобы рефакторинг проверялся счётом, а не верой.

**Developer:** ничего не пишем с нуля, всё собирается из готовых кусков репо: раннер копирует скелет mass_run_creatives (resume, --dry-run, архив перед перезаписью), scorecard копирует smoke_test_creatives (фиксированные заголовки CSV), куратор эталона копирует learn.py (показать → y/n → сохранить с датой и автором). Это 2-3 небольших новых файла, не новая система.

**Trickster:** три ловушки. (1) Живые сайты меняются сами — сайт реально снял пиксель, а стенд орёт «регрессия». Значит, сравнение обязано отличать «сканер ослеп» от «сайт изменился» (**дрейф**) — по сырым network-запросам, которые сканер и так пишет. (2) Стенд может сам врать — его пробы улик должны быть НЕЗАВИСИМЫ от продакшн-кода детекции, иначе сломанный рефакторинг ослепит и стенд. (3) Не дать стенду сожрать всю неделю — он инструмент, а не продукт; полный прогон 10 сайтов ≈ 60–80 минут, поэтому нужен `--fast` режим (2 маленьких сайта, ~10 минут) перед каждым коммитом, полный — фоном/на ночь.

**Client (лид TNC, который получит отчёт):** для питча страшнее всего **false positive** — если в отчёте написано «у вас есть Snapchat-пиксель», а его нет, доверие к агентству умирает на месте. Пропуск (miss) — неприятно, но починимо. Поэтому в эталоне и в фиксах приоритет: сначала убить фабрикации (substring-матчи, пустые ID), потом добирать пропуски.

## Что делаем (рекомендованный подход)

### 1. Золотой корпус — папка `01-client-discovery/engine/golden/` (коммитится; `scans/` в .gitignore, туда нельзя)

11 сайтов из имеющихся 17 (разные платформы/языки/размеры, 46 страниц суммарно):

| Домен | Платформа | Зачем |
|---|---|---|
| tinytronics.nl | opencart, 2 стр. | быстрый эталон (`--fast`) |
| allbirds.com | shopify | бывший Snapchat/Bing false positive |
| bombas.com | shopify | кейс 404 + гео-модал |
| fritz-kola.de | shopify, DE | кейс редирект-слепоты |
| gymshark.com | shopify | дубль Meta; Snapchat в GTM |
| pipsnacks.com | shopify | Pinterest-пропуск; Cal.com FP |
| artbouquet.shop | shopify, 7 стр. | **POST-слепота парсера событий** (BUGS-2026-07-13, коммит 6b9a4ec): 6 из 7 страниц — false GAP, Meta AddToCart/InitiateCheckout реально летят POST'ом |
| garage.suspenair.fr | wordpress, FR | второй fast-домен |
| studioaplus.ca | wordpress | средний WP |
| nissan.ie | generic | кейс unverified-статуса |
| plurio.ai | framer | SaaS/generic |

**Два ключевых решения:**
- **step1 замораживается**: копируем существующие `<domain>_step1.json` в `golden/` и всегда гоняем step2 по одному и тому же списку URL. Иначе живой sitemap меняется и счёт шумит. Обновление — только сознательное (`--refresh-step1`).
- **Эталон = правда, а не текущий вывод сканера.** Для 5 Shopify-сайтов правда уже записана руками в `scans/_shakedown_2026-07-08/README.md` (fritz-kola редиректится, pipsnacks реально шлёт Pinterest…). Поэтому в день 3 стенд честно покажет ~80–90% с поимённым списком известных провалов — и всю неделю число растёт к 100% по мере фиксов. Прогресс виден.

**Формат `golden/expected_<domain>.json`** — проверяются ТОЛЬКО стабильные поля: нормализованный `status` каждой страницы (OK / GAP / NO_TRACKING / NO_CTA / UNVERIFIED + новые REDIRECTED / HTTP_ERROR), набор обнаруженных платформ, `has_cta`, `missing_events`, счётчики по сайту, платформа сайта, GTM-платформы. Волатильное (тексты кнопок, порядок списков, сырые запросы) — никогда. Отсутствующее в эталоне поле = не проверяется. Плюс провенанс: `verified_by`, `verified_date`, `scanner_commit`, `notes`.

Куратор `make_expected.py` (по образцу learn.py): читает свежий step2.json, показывает по странице по-русски («/, homepage, статус OK, платформы Meta+GA4 — верно? y/n/правка»), пишет эталон с провенансом. `--update` — точечная перезаверка после дрейфа.

### 2. Стенд — `eval_run.py` + `eval_lib.py`

На каждый домен корпуса: прогнать `step2_scan.run()` по замороженному step1 → сравнить с эталоном → вердикт по каждой проверке:

- **MATCH** — совпало.
- **FAIL** — расхождение, и сырые network-запросы страницы (сканер их и так сохраняет) доказывают, что сканер ослеп или сфабриковал. Это регрессия — валит прогон (exit 1).
- **DRIFT** — расхождение, но улик в сырых запросах нет → похоже, сайт сам изменился. Жёлтая пометка «перепроверь глазами и обнови эталон», прогон не валит.

Улики ищутся по маленькому независимому списку проб (`facebook.com/tr`, `ct.pinterest.com`, `bat.bing.com`…) — специально НЕ из продакшн-реестра.

**Пробы обязаны видеть POST, не только URL** (поправка из параллельного чата, факт проверен): сейчас сырьё — это `request_urls_all[:300]`, только строки URL (generic_scanner.py:31,96). Для проверки «платформа была?» URL хватает (URL Meta-POST'а — тот же `facebook.com/tr`), но для проверки «событие AddToCart летело?» нужно тело — Meta шлёт содержательные события multipart-POST'ом, TikTok вообще всё в JSON-телах (BUGS-2026-07-13). Поэтому маленькое аддитивное изменение в трёх сканерах: рядом с `network_requests` сохранять `pixel_hits` — записи `{url, method, body_snippet}` только для известных пиксель-хостов (чтобы не раздувать JSON). Пробы стенда на уровне событий матчат по методу+телу. Без этого стенд унаследует ту же слепоту и не отличит FAIL от DRIFT на POST-классе багов.

Вывод: `scans/_eval/<timestamp>/scorecard.md` + `.csv` (таблица домен → N/M → FAIL/DRIFT поимённо) и строка в коммитируемый `golden/history.csv` — история счёта по времени = **кривая доверия** (`eval_run.py --history`).

Флаги: `--fast` (tinytronics + garage.suspenair.fr, ~10 мин, перед каждым коммитом), `--domains`, `--skip-scan` (только пересравнить без пересканирования — для отладки), `--refresh-step1`. Полный прогон (46 страниц ≈ 70–90 мин) — фоном или на ночь. CI не делаем.

### 3. Единственный рефакторинг недели — `platforms.py`

Один реестр: на платформу — network-паттерны (регэкспы с границами слов, не подстроки), GTM-сигнатуры, Shopify-маркеры и app-ID, tier1/tier2/noise события. Конструктор реестра кидает исключение на дубликат ключа (класс Hotjar-бага умирает навсегда).

**Миграция в два шага — принципиально:**
- **Шаг A (эквивалентность):** старые имена (`PIXEL_RULES` и т.д.) становятся производными от реестра; pytest сверяет их с замороженными копиями старых таблиц; счёт стенда обязан остаться идентичным. Перед шагом — `git tag snapshot-before-merge`.
- **Шаг B (контент):** добавить Pinterest везде (баг B5), Snapchat в GTM-сигнатуры (B6), слить Hotjar-дубль, перевести матчинг на границы слов (убивает Cal.com-класс, баг A3), проверить пустой metaPixelId (A4). Ожидание на стенде: известные FAIL гаснут, новых ноль.

### 4. Два дешёвых системных фикса

Один хелпер `navigate_and_gate()` в base_scanner (заменяет три почти одинаковых goto-блока в generic/shopify/wordpress-сканерах): после перехода сравнить финальный URL с запрошенным и проверить HTTP-статус. Статус ≥400 → страница помечается `HTTP_ERROR` и не аудируется (баг C9); смена хоста или схлопывание пути в `/` → `REDIRECTED` (баг C8, кейс fritz-kola «просканировали главную трижды под видом cart/contact/product»). В step2 — ранняя ветка «gate-статус не перезаписывается» + счётчики, в report.py — минимальный рендер.

### 5. Процесс-петля вместо whack-a-mole — `engine/PROCESS.md`

1. Новый клиентский сайт → обычный скан.
2. Триаж по чек-листу: лишняя платформа? пропущенная? неверный статус? редирект/404? мусорные кнопки?
3. Каждый подтверждённый промах = **два артефакта обязательно**: фикс кода + запись в эталоне, воспроизводящая баг (новый сайт в корпус или ассерция в существующий expected).
4. Перед коммитом: pytest + `eval_run.py --fast`. Полный прогон — в тот же день фоном. Коммит только на зелёном (0 FAIL; DRIFT допустим с пометкой).
5. Дрейф: подтвердил глазами → `make_expected.py --update` с новой датой.

**Первые входы в петлю после зелёного стенда** (фиксируются в PROCESS.md, на этой неделе НЕ чинятся):
- **6 багов из BUGS-2026-07-13** (POST-парсер событий, клик-события в отчёте, «не проверялось» ≠ «отсутствует», кликер на Shopify-листингах, journey cart→checkout, Pinterest в PIXEL_RULES). Проблемы 1–2 уже в работе в параллельном чате — эталон artbouquet записывает POST-правду (Meta AddToCart/InitiateCheckout, TikTok-события), поэтому до приземления того фикса стенд честно показывает FAILs на artbouquet, а после — верифицирует фикс автоматически.
- **B7** из shakedown (GTM через network — улики уже лежат в network_requests).

**Критерий доверия:** 3 новых сайта подряд данного класса без единого дефекта кода при счёте ≥95%. Shopify достигнет первым.

## Роадмап: 7 дней × 2 часа (каждый день кончается чем-то запускаемым)

| День | Работа | Проверка в конце дня |
|---|---|---|
| **1** | План → в репо: `engine/TESTBED-PLAN.md` + указатель из CLAUDE.md; `golden/` + заморозка step1 (копия 11 файлов); `eval_lib.py` (нормализация статусов, сравнение — чистые функции + pytest); `make_expected.py`; эталоны tinytronics + garage.suspenair.fr | pytest зелёный; 2 эталона в git |
| **2** | `eval_run.py`: прогон, диффер, scorecard, history.csv, флаги; `pixel_hits` (url+method+body_snippet для пиксель-хостов) в трёх сканерах — аддитивно | `eval_run.py --fast` печатает таблицу N/M |
| **3** | Эталоны остальных 9 сайтов (правда — из shakedown README + отчётов; artbouquet — POST-правда из BUGS-2026-07-13); первый полный прогон | Baseline в history (~80–90%), known-fails поимённо |
| **4** | `platforms.py` шаг A (эквивалентность) + тесты; `git tag snapshot-before-merge` | Счёт идентичен дню 3; pytest зелёный |
| **5** | Шаг B: Pinterest, Snapchat-GTM, Hotjar-дубль, границы слов, metaPixelId | Известные FAIL → MATCH, новых ноль, счёт вырос |
| **6** | `navigate_and_gate()` в трёх сканерах + step2 + report | fritz-kola/bombas FAIL → MATCH |
| **7** | PROCESS.md; **учение**: 1 свежий сайт через полную петлю руками; парк-лист; буфер | История показывает рост; петля пройдена один раз |

Судьба 12 shakedown-багов: **чинятся на неделе** A3, A4, B5, B6, C8, C9 (A1/A2 уже починены ранее). **Паркуются** B7 (GTM через network — первый кандидат ПОСЛЕ недели, улики уже в network_requests), C10 (гео-модал), D11 (нелокализованные SKIP_TEXTS кнопок), D12 (косметика статусов). **6 багов BUGS-2026-07-13 — вне скоупа недели** (записаны в PROCESS.md как первые входы в петлю; Проблемы 1–2 параллельно чинятся в другом чате, стенд их верифицирует через эталон artbouquet).

## Что НЕ делаем (явно)

- **Никакого rewrite** — не подтверждён фактами, ядро здоровое.
- **Google Transparency Center — вне скоупа недели.** Добавлять после зелёного стенда, тем же методом (свой корпус, свой эталон).
- Не объединяем два кнопочных детектора и не чиним их англоязычность — это СЛЕДУЮЩИЙ рефакторинг, под уже готовым стендом.
- Никакого CI / pre-commit-хуков — гейт ручной.
- Не трогаем learn.py / patterns.json / EXTERNAL_SERVICES-таксономию.

## Риски

- **Дрейф живых сайтов** — закрыт вердиктом DRIFT + перезаверкой; в корпусе стабильные бренды.
- **70–90 мин полного прогона** — фон/ночь + `--fast`; `--max-pages` в стенде не использовать (искажает счётчики).
- **`step2_scan.run()` делает `sys.exit(1)` на отсутствующем файле** — раннер обязан пре-чекать пути.
- **cp1252** — каждый новый entry point начинается с `setup_console()`.
- **Сползание 2-часовых дней** — день 7 наполовину буфер; стенд полноценен уже с 6 доменами.

## Файлы

**Изменяемые:** `scanners/base_scanner.py` (производные таблицы из реестра; navigate_and_gate; границы слов в матчинге), `scanners/generic_scanner.py` / `shopify_scanner.py` / `wordpress_scanner.py` (goto-блок → хелпер; маркеры из реестра; `pixel_hits` рядом с `network_requests`), `gtm_analyzer.py` (сигнатуры из реестра), `step2_scan.py` (gate-статусы + счётчики), `report.py` (рендер новых статусов, минимально), `CLAUDE.md` (указатель на TESTBED-PLAN.md).

## Верификация

- Дни 1–2: pytest по eval_lib + `eval_run.py --fast` руками.
- День 3: полный прогон, baseline-счёт записан, known-fails сверены глазами с shakedown README.
- Дни 4–6: после каждого шага — `--fast` + pytest; полный прогон в конце дня; счёт не падает, известные FAIL гаснут.
- День 7: живое учение — один свежий сайт через всю петлю (скан → триаж → фикс или парк → эталон → зелёный стенд → коммит).
- Перед каждым merge/коммитом — `git tag snapshot-before-merge` (правило CLAUDE.md).
