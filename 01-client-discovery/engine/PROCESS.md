# PROCESS — петля развития сканера под стендом

Стенд построен 2026-07-13 (неделя, TESTBED-PLAN.md). Это правила игры ПОСЛЕ:
как чинить, добавлять сайты и не съезжать обратно в whack-a-mole.

## Правило №1 — гейт Rodion'а (не обсуждается)

**Любая правда записывается только после явного «да» Rodion'а.** Правда =
события, пиксели, теги, кнопки и их кликабельность, куки и их принятость —
всё видимое на экране. Вопрос — с уликой свидетеля, по одному сайту,
полными предложениями, в его терминах («пиксель» = Meta/TikTok — проверяемо
Pixel Helper'ом; у Google — «теги» — проверяемо Tag Assistant). На гейт
выносится только то, что он может проверить своими инструментами; протокольные
машинные факты (HTTP-код) фиксируются с пометкой источника. Правила отсутствия
(platforms_forbidden) — ТОЛЬКО после его личной проверки: «свидетель не видел
запросов» ≠ «пикселя нет» (урок bombas/Pinterest).

## Петля нового клиентского сайта

1. Обычный скан: `step1_sitemap.py <домен>` → `step2_scan.py` → `report.py`.
2. **Триаж по чек-листу:**
   - лишняя платформа в отчёте? (фабрикация — худший класс, ломает доверие питчу)
   - пропущенная платформа? (сверься с witness/Pixel Helper)
   - неверный статус? (GAP там, где unverified; OK там, где red flag)
   - редирект / 404 / пустая страница в выборке?
   - мусорные кнопки в CTA / кликнуто что-то опасное?
3. Каждый подтверждённый промах = **ДВА артефакта, обязательно**:
   (а) фикс кода, (б) запись в эталоне, воспроизводящая баг — либо сайт
   добавляется в корпус, либо в существующий эталон дописывается ассерция/пин.
4. Сайт интересный (новый класс/платформа/баг) → в корпус (см. ниже).

## Новый сайт в корпус

1. `corpus.json` += домен (платформа, язык, «зачем он в корпусе»).
2. `python eval_run.py --refresh-step1 <домен>` — свежий step1 в golden/
   (раздутый to_scan усечь: главная + все priority-1 + по одной странице
   каждого типа; пометка `_testbed_note` в файле).
3. `python witness.py <домен> --pages` (+ `--journey` для e-commerce) —
   witness-прогон ОБЯЗАТЕЛЕН: без независимых улик эталона не бывает.
4. Свежий step2 по замороженному step1 → `make_expected.py <домен> --draft`.
5. **Гейт**: правда-факты с уликами → «да»/«нет» Rodion'а → schema v2
   + `verified_via` (что показывали, когда подтвердил).
6. `python witness_check.py <домен>` → 0 противоречий.
7. `python eval_run.py --domains <домен>` → зелёный (или known fails поимённо).
8. Коммит: эталон + witness-JSON + facts вместе.
**Эталон без verified_via не коммитится.**

## Перед КАЖДЫМ коммитом кода сканера

```bash
python -m pytest -q            # все test_*.py (пины багов; ~секунда)
python eval_run.py --fast      # 2 fast-домена, ~3 мин
```
Полный `python eval_run.py` — в тот же день (фоном, ~50 мин). Коммит только
на зелёном: **0 FAIL**; DRIFT допустим с пометкой. `git tag -f
snapshot-before-merge` перед коммитом (правило CLAUDE.md). Строку history.csv
коммитить вместе с фиксом — кривая доверия должна расти.

## DRIFT-протокол (жёлтое в scorecard)

DRIFT = расхождение без улик против сканера: возможно, сайт сам изменился.
1. Открой witness-файл/скриншот рядом с эталоном, проверь глазами (или
   Pixel Helper'ом — 30 секунд).
2. Сайт реально изменился → `make_expected.py <домен> --update` (пины
   переносятся автоматически) + новая verified_date. Через гейт.
3. Мигающий 3rd-party (LinkedIn на bombas) → пометка в notes эталона,
   ассерцию ослабить или убрать.

## Критерий доверия (когда сканеру можно верить)

По классу платформ: **3 новых сайта подряд** этого класса, где триаж не нашёл
НИ ОДНОГО дефекта кода (только пополнение эталона), при полном eval ≥ 95%
MATCH (DRIFT не в счёт). Shopify достигнет первым (7 сайтов в корпусе).
Google TC добавлять ТОЛЬКО после этого — тем же методом (свой корпус,
свой свидетель, свои эталоны).

## Команды-шпаргалка

```bash

python eval_run.py                     # полный прогон, пишет history
python eval_run.py --history           # кривая доверия
python eval_run.py --domains a.com     # точечно
python eval_run.py --skip-scan         # только дифф, без сканов (отладка; history не пишет)
```
