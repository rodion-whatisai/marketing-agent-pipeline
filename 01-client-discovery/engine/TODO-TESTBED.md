## Метод стенда / гейта

1. **«Нет запросов у свидетеля» ≠ «пикселя нет».** bombas: свидетель — ноль
   Pinterest-запросов, Pixel Helper Rodion'а видит тег 2612751672311. Правила
   отсутствия (platforms_forbidden) — ТОЛЬКО после его личной проверки. Применено
   с гейт-раунда №2.
2. **В гейт-вопросы — только проверяемое его глазами/инструментами.** SHA-хэши
   и сырые protocol-факты не выносить на его валидацию («это твоя ошибка»).
   Машинные факты (HTTP-код) можно фиксировать без гейта, с пометкой источника.
3. **Правило второго пикселя:** факт наличия второго пикселя где бы то ни было
   на сайте регистрируем всегда; события на неприоритетных страницах
   (store-locator и т.п.) — не важны.
4. **Непонятный пункт без ссылок и инструкции — не на гейт.** Нужны адреса
   страниц и шаги («открой X, посмотри Y»), иначе — в разбор.

## Сканер / свидетель

5. **Consent-reload false-negative** (bombas): Meta-пиксель появился только
   ПОСЛЕ перезагрузки страницы после принятия куки. Сканер/свидетель могут
   давать ложное «пикселя нет». Нужна перезагрузка после consent-клика.
6. **Пиксель вне CMP** (gymshark): до принятия куки виден один Meta-пиксель,
   после — два. Один пиксель не подключён к согласию. Новый класс проверки.
7. **Checkout-проход** (allbirds/bombas/gymshark/pipsnacks/artbouquet): Shopify
   checkout — отдельная среда, пиксели «исчезают» для Pixel Helper; при этом
   клики там дают AddPaymentInfo/purchase-подобные события БЕЗ ввода данных
   (misconfig-класс). Нужен отдельный описанный проход и правила ярлыков;
   «AddToCart без Purchase не бывает» — проверять парой.
8. **Порядок баннеров** (fritz-kola): куки-баннер лежит ПОВЕРХ гео-баннера —
   кликать надо куки первым; у нас порядок гео→куки. Проверить popup_handler.
9. **Thank-you после сабмита** (studioaplus): пустая форма принимается и
   показывает «Thank you» — детект thank-you-состояния после сабмита как
   сигнал (мы формы не сабмитим — но идея зафиксирована).
10. **LinkedIn: network vs Helper** (nissan): collect-запросы в трафике есть,
    Pixel Helper тег не видит. Разобраться, кому верить и как ярлычить.
11. **Отбор страниц:** contact-страница не нужна для e-commerce аудита —
    правило в классификатор/выборку. Confirmation-страницы форм (nissan) при
    прямом заходе редиректят на формы — в выборку не годятся.
12. **fritz-kola «главная»** = jobs.fritz-kola.de (артефакт sitemap-fallback) —
    refresh-step1 на EN-версию fritz-kola.com/en (связано с EN-first веткой).

## Архитектура (идея Rodion'а, 2026-07-13)

16. **Платформенные правила чтения сигналов.** Сбор сигналов уже «максимум» и
    уже разведён по платформам (три сканера, routing по platform_detector);
    ИНТЕРПРЕТАЦИЯ же (статусная лестница OK/GAP/…) — общая для всех. Идея:
    одни и те же сигналы читать по-разному в зависимости от платформы.
    Первые правила из гейт-раундов: (а) Shopify: чекаут — закрытая среда,
    «AddToCart есть, дальше тишина» ≠ дыра; AddToCart двоится (продукт+корзина);
    пиксель может жить в web-pixel сэндбоксе, невидимом для Tag Assistant;
    (б) WordPress: конверсия часто в iframe-форме (Jotform/Calendly) — ноль
    сетевых событий это «не настроено», не «сломано»; (в) кастом: только прямые
    сигналы, ничего не предполагаем. Эталоны остаются per-site (правда всегда
    о конкретном сайте); платформенными становятся ПРАВИЛА ярлыков. Дом для
    этого — platforms.py (реестр дней 5–6); внедрять после недели, по одному
    правилу через петлю, под стендом. Схема: scanner-decision-flow (чат 07-13).
