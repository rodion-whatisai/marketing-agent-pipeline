# Дневной контроль бюджета — как медиаплан раскладывает деньги по дням

> Разбор одного механизма из рабочего медиаплана: как месячный бюджет ад-сета
> распределяется по дням так, чтобы деньги шли туда, где лид дешевле. Формулы и сэмпл
> реверс-инжинирены из воркбука **«Mediaplans 2025»** (лист `Mediaplan Dallas`,
> ад-сет `FB_IG / DAL / Lead Gen / Wide`). Это планинговая механика: она готовит
> *целевые* дневные числа, против которых блок 03 потом сверяет факт.

Вводим термины по ходу — «сэйв спейс».

- **Ад-сет** (ad set) — одна настроенная связка «аудитория + цель + плейсмент» внутри
  кампании. У нас он опознаётся ключом-склейкой `FB_IG+City+Objective+Targeting+Advantage+Language+BD-Month`.
- **Бенчмарк** — усреднённый факт прошлого периода (CPM, CTR, CVR, CPL) по этому ад-сету.
  Берётся из листа `Benchmarks table`.
- **CPL** (cost per lead) — цена одного лида = потрачено ÷ лиды.
- **Период** — месяц нарезан на 4 куска: календарная неделя 1, 2, 3 и «остаток месяца».
  У каждого периода **свой** бенчмарк, потому что в начале и в конце месяца реклама ведёт
  себя по-разному.
- **Пейсинг** (pacing) — как сумма размазывается по дням внутри месяца.

---

## 1. Принцип одной фразой

> Месячный бюджет ад-сета **фиксирован**. Каждый день забирает себе **бенчмарк CPL того
> периода**, в который попадает. Дальше бюджет растекается по дням **обратно пропорционально
> CPL**: где лид дешевле — туда денег больше. Сумма по дням всегда сходится ровно в месячный
> бюджет.

То есть это не «бюджет ÷ число дней» поровну. Это «лей больше в дни, где лид дешёвый».

---

## 2. Откуда берётся месячный бюджет (воронка плана)

Сначала из бенчмарка считается весь месяц — простая воронка. Буквы в скобках — это колонки
листа `Mediaplan Dallas`.

```
Бюджет (S)   = Impressions_target × CPM / 1000      # стандартная CPM-математика
Клики  (W)   = CTR × Impressions
Лиды   (Z)   = Клики × CVR                          # CVR = доля кликов, ставших лидом
CPL    (AA)  = Бюджет / Лиды
```

Дальше воронка достраивается до денег (для ROAS-картинки):

```
Брони/Arrivals = Лиды × arrival_rate    # arrival_rate тянется INDEX/MATCH'ем по городу
                                         # из листа 'Arrivals based planning'
Выручка        = Arrivals × средний чек (Av.cheque, тут $2000)
ROAS           = Выручка / Бюджет
```

Для нашего сэмпла месяц получился: **бюджет $3056.76 → ~237 лидов → CPL ~$12.89**.

---

## 3. Формула дневного пейсинга (ядро)

Дневной бюджет каждого дня `d`:

```
budget_d = monthly_budget × (1 / CPL_d) / Σ_i (1 / CPL_i)
                              └─ вес дня ─┘   └ сумма весов по всем дням месяца ┘
```

- `CPL_d` — бенчмарк CPL периода, в который попал день `d`.
- `1 / CPL_d` — «вес» дня. Чем дешевле лид, тем больше вес.
- Делим на сумму всех весов → нормировка. Поэтому `Σ budget_d` == `monthly_budget`, копейка
  в копейку.

Дневные лиды — просто:

```
leads_d = budget_d / CPL_d
```

Так выглядит **настоящая** формула ячейки дневного бюджета (день 1, `AL2`) — массивная,
суммирует по всей строке дневных CPL:

```excel
=IFERROR( IF( CX2>0,
              (1/CX2) / SUM( IF( ($CX2:$EB2<>"")*($CX2:$EB2>0), 1/$CX2:$EB2, 0) ) * $S2,
              0), "")
```

где `CX2:EB2` — это 31 ячейка дневных CPL, а `$S2` — месячный бюджет.

---

## 4. Как день узнаёт свой CPL (привязка к периоду)

Дневной CPL (`CX2`) не вписан руками — он **выбирается** из `Benchmarks table` по двум
условиям: совпал ключ ад-сета И день попал в окно `[From, To]` строки-бенчмарка.

```excel
=FILTER( 'Benchmarks table'!CPL,
         'Benchmarks table'!Key  =  $M2,          # тот же ад-сет
         'Benchmarks table'!From <= день,          # начало периода
         'Benchmarks table'!To   >= день )          # конец периода
```

Вот почему дневной бюджет «ступеньками» меняется на границах недель: день перепрыгивает в
следующий период → подтягивает другой CPL → меняется его вес.

## 5. Дата-сэмпл (реальные числа из листа)

Ад-сет `FB_IG / DAL / Lead Gen / Wide / Not Advantage / EN`, июль 2025.
Полная подневная раскладка — в [daily-budget-sample.csv](daily-budget-sample.csv) (30 дней).
Сжато по периодам:

| Период | Дней | CPL бенчмарк | Бюджет/день | Бюджет периода | Лиды |
|---|---:|---:|---:|---:|---:|
| Календарная неделя 1 | 7 | $14.25 | $102.57 | $717.97 | 50.4 |
| Календарная неделя 2 | 7 | $13.39 | $109.15 | $764.08 | 57.1 |
| Календарная неделя 3 | 7 | **$10.19** | **$143.43** | $1 004.03 | 98.5 |
| Остаток месяца | 9 | $23.05 | $63.41 | $570.68 | 24.8 |
| **Итого** | **30** | — | — | **$3 056.76** | **230.7** |

Читать так:
- **Неделя 3 — самый дешёвый лид ($10.19) → самый жирный дневной бюджет ($143).** Дорогой
  «остаток месяца» ($23.05) получает меньше всех ($63). Это и есть «1/CPL»-принцип в цифрах.
- Σ дневных бюджетов = **$3 056.76** = месячный бюджет ровно (нормировка сошлась).
- Σ дневных лидов = 230.7 против ~237 в помесячной воронке: маленький зазор от того, что
  дневные лиды считаются как `бюджет/CPL_периода`, а помесячные — через CVR-ветку, плюс в
  этом ад-сете заполнено 30 дней, не 31.

## 6. Что отсюда берёт блок 03

- **Целевые дневные числа.** `budget_d / leads_d / CPL_d` по ад-сету на каждый день — это
  готовый таргет, против которого движок сверяет факт (план vs факт по ключу `Key + дата`).
- **Принцип аллокации.** «Лей в дешёвый лид» (вес `1/CPL`) — это та же идея, что у движка
  при scale/kill, только посчитанная заранее из бенчмарка, а не по живому сигналу.
- **Гранулярность времени.** 4 периода месяца = естественный шаг, на котором меняется
  бенчмарк и, значит, решение по бюджету.
