# 02 · Формирование брифа — скетч подсистемы

> **Статус: спроектировано, в коде ещё нет.** Деталь блока [02 — Planning](README.md):
> как из разговора с клиентом рождается стандартизированный бриф. Псевдокод ниже — схема
> замысла, не рабочий модуль.

## Идея: бриф — это продукт реального времени, не бумажка

Живой брифинг: пока клиент рассказывает, в голове уже складывается, что предложить; потом
берётся время на подумать, рождается идея, клиент с ней соглашается или нет. Здесь то же,
но с двумя сдвигами:

- **думаем сразу** — не уходим думать на два дня; оптика и предложение собираются на ходу,
  с первого же ответа;
- **не доверяем тому, как думает агент → фидбэчим немедленно** — каждый ответ тут же
  проходит детерминированную проверку, а догадки агента возвращаются клиенту на
  подтверждение, не идут как факт.

Поэтому бриф — не форма, которую заполнили и отдали, а **стриминговый сборщик**: ответ за
ответом, и каждый ответ сразу запускает планирование и сразу проверяется.

Стандартный каркас вопросов берётся из индустрии (их в сети много): кто ты · что ты
делаешь · как видишь целевую аудиторию · какой бюджет · какие ресурсы (сайт, креатив…).
Поверх «бумажки» — выцепление инсайтов из разговора. Ключевое требование: **на чтении
самого первого ответа планирование уже началось.**

## Три лица пишут на одной форме, Python — рефери

- **Человек** — клиент / стратег (может быть один; это узкое место). Даёт сырые ответы и
  бизнес-оптику.
- **Агент «зеркало клиента»** — предугадывает ответы клиента, достраивает агентское
  описание бизнеса.
- **Агент «зеркало стратега»** — с первого ответа лепит оптику и предложение и заранее
  видит, какие вопросы понадобятся под выбранный подход / канал.

Эти трое формируют **стандартизированный бриф**. А **Python — это и сама форма (структура
слотов), и рефери**: не пропустит бриф дальше, пока он не полон и не консистентен.
Финальную галку ставит **человек-клиент** (см. гейт ниже).

> Вход стадии — это и есть «минимум три условных лица»: человек-сторона может схлопнуться
> в одного (клиент / стратег), наша сторона — два агента-зеркала. Кратко то же — во входе
> [README](README.md).

## Поток (скетч на уровне кода)

```python
# ── Python: каркас брифа ──────────────────────────────────
brief = Brief(schema=STANDARD_QUESTIONS)   # слоты: кто-ты / что / ЦА / бюджет / ресурсы…
                                           # у слота: required? / «звёздочка» (заменяем pipeline'ом)?
queue = QuestionQueue()                    # очередь вопросов клиенту

# ── Поток: клиент отвечает по одному ответу за раз ────────
for answer in client_stream:
    slot, value = route_answer(answer)     # [Python] в какой слот, нормализовать
    brief.fill(slot, value, src="client")  # [Python] записать + штамп провенанса

    optics = strategist_mirror(brief)      # [АГЕНТ] думаем уже с первого ответа:
    proposal.update(optics)                #         предложение складывается на ходу
    need   = optics.required_inputs()      # [АГЕНТ] что ещё спросить под выбранный подход / канал

    gaps   = check_consistency(brief)      # [Python] противоречия / нехватка — детерминированные правила
    queue.add(gaps + need)                 # [Python] реактивные дырки + упреждающие нужды плана → очередь

    guess  = client_mirror(brief.empty())            # [АГЕНТ] зеркало клиента предугадывает ответы
    brief.fill_provisional(guess, src="guess")       # [Python] вписать как догадку (ждёт подтверждения)

    ask(queue.next())                      # следующий вопрос — пока есть что спросить

# ── Закрытие ─────────────────────────────────────────────
for slot in brief.unanswered_required():
    brief.fill(default(slot), src="assumed")   # [Python] не ответили → допущение + ЯВНЫЙ штамп

while not readiness_gate(brief):           # [Python] полнота + консистентность
    ask(queue.next())                      #          ещё не готов — добираем ответы

if client_validates(brief.standardize()):  # [ЧЕЛОВЕК] клиент подтверждает онбординг — финальная галка
    client_service.handoff(brief)          # → только теперь бриф двинулся дальше
```

## Стор фактов: поток снапшотов (чистый Python, без агентов)

Поток выше — про *взаимодействие*. Это — про то, что из него **остаётся**: факты.

Список вопросов **фиксирован** (приколочен гвоздями; сейчас это реальный бриф TNC,
канонически выводится из планинга позже). Вопросы **не меняем** — **переспрашиваем тот же
вопрос под другим ракурсом** и сохраняем каждый ответ. Цель не «угадать», а **записать за
клиентом и выжать больше**: пошаговое мышление от первого ответа к третьему.

```python
# Фикс-список = поля брифа TNC (immutable; тот же бриф, что в потоке выше, но как сохранённая сессия)
QUESTIONS = ["company", "project", "objectives", "audience", "timeline",
             "product", "benefits", "cta", "budget", "legal",
             "deliverables", "kpis", "acceptance", "extra"]

s = BriefSession(QUESTIONS)

# слепок 1 — клиент заполнил что смог и отправил
s.save_snapshot(first_fill, trigger="client_submit")        # [Python] неизменяемый слепок

# детерминированные триггеры переспроса — бенчмарк / противоречие, как гард $500 vs $700, БЕЗ агента
if benchmark_says_cheaper(s.last("budget"), s.last("kpis")):
    s.attach_followup("budget", angle="KPI достижим дешевле — сэкономить или перевыполнить?")
if big_budget_one_product(s):                                # большой бюджет + одна позиция + голый сайт
    s.attach_followup("budget", angle="не маловато под одну позицию?")

# слепок 2 — клиент ответил на доп-вопросы, пишем в ТОТ ЖЕ стрим
s.save_snapshot(second_fill, trigger="followup_round")       # [Python]

# что остаётся: траектория по вопросу = пошаговое мышление клиента
s.history("budget")
# → [(700k, "как задано", t0), ("сэкономить", "бенчмарк дешевле", t1), ("1.2M", "под одну позицию", t2)]
```

Что этот код **делает** и чего **не делает**:

- **делает (чистый Python):** держит фикс-список; пишет каждую отправку **неизменяемым**
  снапшотом в append-only поток; запускает **детерминированные** триггеры переспроса
  (бенчмарк, противоречие бюджет↔KPI — та же логика, что гард `$500 vs $700`); цепляет тот
  же вопрос с новым ярлыком-ракурсом; отдаёт по каждому вопросу траекторию
  ответ→ракурс→ответ с таймштампами. Снапшоты — **готовые обучающие данные**: видно, как
  клиент менял ответ, когда его переспросили, — то есть как он думает.
- **не делает (это агент, поверх, отдельно):** человеческую формулировку ракурса, новые
  нешаблонные углы, контр-предложение по каналам («рекомендуем другие площадки, дешевле —
  согласен?»). Стор **верно записывает поиск истины, но не ведёт его.**

> **Паркуем:** была идея «живого» брифа — поменял ответ сверху, форма снизу сразу
> перестроилась. Пока неясно, как это надёжно сделать; выбран более простой и ценный путь —
> **меньше фантазировать, больше записывать за клиентом**: фикс-вопросы + поток снапшотов,
> который выжимает больше за счёт переспроса под разными ракурсами.

## Кто что считает

| Python (железобетон) | Агент (суждение) | Человек (гейт) |
|---|---|---|
| схема брифа + слоты (required / «звёздочка») | оптика и предложение «на ходу» (зеркало стратега) | **финальная валидация онбординга** — клиент подтверждает, и только тогда бриф уходит дальше |
| `route_answer` — ответ → слот, нормализация | предугадывание ответов (зеркало клиента) | |
| `check_consistency` — противоречие / нехватка → **флаг**, не суждение | человеческая формулировка вопросов | |
| `QuestionQueue` — копит / дедупит / приоритизирует | предложить новую запись в жёсткий список (→ человек апрувит) | |
| `default` + штамп `assumed` для неотвеченного | | |
| `readiness_gate` — полнота и консистентность | | |
| провенанс на каждом поле | | |

## Провенанс

- **Каждое поле штамповано источником** — `client` / `guess` / `assumed` / `scanned` (из
  блока 01). Видно, где факт, а где догадка.
- **Догадки агента — provisional**: ждут подтверждения клиента, не идут как факт.
- **Допущения всплывают в конце** — бриф явно несёт список «вот это заасьюмлено, не
  подтверждено».
- **Рефери — Python, не второй агент.** Полноту и консистентность решает детерминированное
  правило; финальную галку — человек.

## Гейт: что значит «бриф готов идти дальше»

Две ступени:

1. **Python `readiness_gate`** (необходимое) — все required-слоты заполнены или явно
   заасьюмлены, нет жёстких противоречий.
2. **Человек-клиент** (достаточное) — **валидированный человеком онбординг** = финальная
   отправная точка. Никто, возможно, не будет вычитывать бриф целиком (или перестанет с
   какого-то момента) — поэтому в рамках онбординга подтверждение клиента и есть та галка,
   после которой бриф двигается в клиентский сервис.

> Это сознательно **самый простой вариант** на сейчас; дальше гейт может усложниться —
> дверь открыта.

## Пометки на полях (паркуем)

- **Креативы — отдельная история.** «Звёздочка»-слот (дизайн / креатив отсутствует)
  делегируется в отдельный конвейер креативов; внутрь здесь не лезем.
- **«Звёздочка»-слоты** (нет сайта / нет лендинга) → детерминированный роутер ставит
  задачу в под-pipeline «создать сайт / лендинг». Сам pipeline — отдельно.
- **Курица / яйцо** — что у клиента раньше: ресурс (сайт) или инсайт «как сайт должен
  выглядеть». Не замыкаем.
- **Теневой тест жёстких правил** на «чужих» воронках (гоняем правило там, где оно не
  заточено, смотрим — сработает ли) — 4–5й приоритет; механизм детерминированный,
  кандидатов предлагает агент.
